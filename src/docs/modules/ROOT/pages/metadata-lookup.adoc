MyMediaScanner uses a tiered metadata lookup strategy to resolve a scanned barcode into rich item metadata.
The strategy prioritises cached results, falls back through specialist APIs by media type, and ultimately defers to UPCitemdb if no specialist source can satisfy the request.

== Lookup Order

The lookup follows five ordered steps:

=== 1. Barcode Cache Check

Before making any network request, the use case queries the `barcode_cache` SQLite table.
If a record exists and its `cached_at` timestamp is fewer than seven days old, the cached raw response is returned immediately and mapped to a `MetadataResult` domain entity.
This avoids redundant API calls for recently scanned items and works fully offline for repeat scans.

=== 2. Barcode Type Detection

When no usable cache entry exists, the barcode string is analysed to determine its type:

[cols="1,2", options="header"]
|===
| Detected Type
| Pattern / Rule

| IMDb ID
| Begins with `tt` followed by digits (e.g. `tt0111161`)

| ISBN-13
| 13-digit EAN beginning with `978` or `979`

| ISBN-10
| 10-character string with optional `X` check digit

| EAN-13
| 13-digit code not matching ISBN prefix

| UPC-A
| 12-digit code
|===

Detection logic lives in `core/utils/barcode_utils.dart`.

=== 3. Type-Routed API Lookup

The detected type, combined with any media type hint supplied by the user on the scan screen, determines which specialist API client is called first.

[cols="2,2,2", options="header"]
|===
| Media Type
| Barcode Standard
| Primary Source

| Film / TV
| EAN-13, UPC-A
| TMDB

| Music
| EAN-13, UPC-A
| MusicBrainz → Discogs (fallback)

| Books
| ISBN-10, ISBN-13
| Google Books → Open Library

| Video Games
| EAN-13, UPC-A
| UPCitemdb (IGDB planned for v2)
|===

Routing rules:

* An **IMDb ID** (`tt*`) bypasses the type hint and calls the TMDB `/find/\{external_id}` endpoint directly.
* An **ISBN** is sent to Google Books first; if no result is returned, Open Library is tried as a fallback.
* An **EAN-13 or UPC-A** with a film or TV hint is routed to TMDB search.
* An **EAN-13 or UPC-A** with a music hint is routed to MusicBrainz first.
If MusicBrainz returns nothing, is rate-limited, or gives only low-quality data, the repository falls back to Discogs and finally UPCitemdb.
* An **EAN-13 or UPC-A** with a game hint is sent directly to UPCitemdb.

For MusicBrainz matches, the repository ranks candidates by status (Official preferred), format (CD, vinyl, and cassette preferred over digital), completeness (label, catalogue number, track count, date, and country) and the MusicBrainz server score.
A clearly dominant match is auto-accepted; otherwise the user sees the disambiguation screen with country, label, catalogue number, track count, and status on each candidate card.

MusicBrainz enforces strict rate limits.
A dedicated `RateLimitAwareClient` throttles requests to approximately 1 req/s and, on HTTP 503, enters a cooldown period during which non-critical follow-up calls are skipped.

=== 4. UPCitemdb Fallback

If the specialist API returns no results, the use case falls back to UPCitemdb.
UPCitemdb is a general-purpose product database and can resolve many retail barcodes even when specialist sources lack coverage.
It is also the primary source for video games until IGDB integration is added in a later release.

=== 5. Cache and Map

Once a raw API response is obtained, it is:

. Persisted to the `barcode_cache` table with the current timestamp.
. Passed through the appropriate mapper (e.g. `TmdbMapper`, `DiscogsMapper`, `GoogleBooksMapper`) to produce a `MetadataResult` domain entity.
. Returned to the calling use case for presentation.

== API Clients

Each external API is represented by a dedicated client class backed by its own Dio instance.
`TmdbApiClient`, `DiscogsApiClient`, `GoogleBooksApiClient`, `OpenLibraryApiClient`, and `UpcItemDbApiClient` are Retrofit-generated clients.
`MusicBrainzApi` and `CoverArtArchiveApi` are manual Dio clients (Retrofit is not used for these two, as their response shapes benefit from hand-written parsing).

[cols="2,3", options="header"]
|===
| Client Class
| Endpoint Base URL

| `TmdbApiClient`
| `https://api.themoviedb.org/3`

| `MusicBrainzApi`
| `https://musicbrainz.org/ws/2`

| `CoverArtArchiveApi`
| `https://coverartarchive.org`

| `DiscogsApiClient`
| `https://api.discogs.com`

| `GoogleBooksApiClient`
| `https://www.googleapis.com/books/v1`

| `OpenLibraryApiClient`
| `https://openlibrary.org`

| `UpcItemDbApiClient`
| `https://api.upcitemdb.com/prod/trial`
|===

TMDB and UPCitemdb require user-supplied API keys read at runtime from `SecureStorageService` via a Riverpod provider.
Discogs is optional — a key enables it as a music fallback, but the app functions without one.
MusicBrainz, Cover Art Archive, Google Books, and Open Library need no keys.
Users configure their keys in the Settings screen; keys are never bundled with the application.

== Metadata Fields

=== Common Fields

All media types expose the following fields on `MetadataResult`:

* `title` — primary title string
* `subtitle` — optional secondary title or edition label
* `coverImageUrl` — URL of the primary cover or artwork image
* `year` — release or publication year
* `genres` — list of genre strings
* `description` — synopsis or back-cover text
* `publisher` / `label` / `studio` — releasing organisation (field name is type-specific)
* `format` — physical format (e.g. Blu-ray, CD, Paperback)
* `barcode` — the original barcode string

=== Film and TV

[cols="2,3", options="header"]
|===
| Field
| Description

| `director`
| Primary director name(s)

| `cast`
| Lead cast members list

| `runtime`
| Runtime in minutes

| `rating`
| Age / content rating (e.g. PG-13, 12A)

| `tmdbId`
| TMDB numeric identifier
|===

=== Music

The following fields are populated on `MetadataResult` for music items.
Fields sourced from MusicBrainz or Cover Art Archive are stored in `extraMetadata` rather than dedicated columns, as they may not be present for every release.

[cols="2,3", options="header"]
|===
| Field
| Description

| `artists`
| Artist or band name(s)

| `trackListing`
| Ordered list of track titles with durations

| `label`
| Record label name

| `catalogueNumber`
| Label catalogue reference

| `musicbrainzReleaseId`
| MusicBrainz release MBID (UUID); stored in `extraMetadata`

| `musicbrainzReleaseGroupId`
| MusicBrainz release-group MBID; stored in `extraMetadata`

| `musicbrainzArtistIds`
| List of artist MBIDs for all credited performers; stored in `extraMetadata`

| `releaseDate`
| Full release date string (e.g. `1994-09-26`) where available; stored in `extraMetadata`

| `releaseCountry`
| ISO 3166-1 alpha-2 country code of the release (e.g. `GB`); stored in `extraMetadata`

| `packaging`
| Physical packaging description (e.g. `Jewel Case`, `Gatefold`); stored in `extraMetadata`

| `trackCount`
| Total track count across all discs; stored in `extraMetadata`

| `discCount`
| Number of discs in the release; stored in `extraMetadata`

| `status`
| MusicBrainz release status (e.g. `Official`, `Promotional`); stored in `extraMetadata`

| `discogsReleaseId`
| Discogs release numeric identifier; only present when the Discogs fallback fired; stored in `extraMetadata`
|===

==== Cover Art

Cover art is fetched from the Cover Art Archive by release MBID, with release-group MBID as a fallback when the specific release has no dedicated front image.
When neither MBID yields a front image, the repository preserves whatever cover URL other fallback providers (Discogs, UPCitemdb) supplied.

=== Books

|===

| `googleBooksId`
| Google Books volume identifier
|===

== Disambiguation

When an API returns multiple candidate matches for a single barcode (common with EAN codes shared across regional releases), the use case emits a list of `MetadataCandidate` objects.
The router navigates to the disambiguation screen (`/disambiguation`), where the user selects the correct release.
The chosen candidate is then passed back through step 5 (cache and map) as if it were a single result.
